## 12:57 — [Audio] Fix: realtime mic occasionally deaf (no input buffers on re-wake)
- Symptom (from ~/.iris/iris.log): on some re-wakes the realtime engine logs "audio started" +
  "mic streaming started" but NO "first buffer" line ever appears, so IRIS sits deaf for 15s then
  idle-pauses. Root cause: AVAudioEngine input intermittently delivers no buffers when the
  wake-word engine is torn down and the realtime engine is started on the same input device before
  it's ready (same family as the `-10877` AudioUnit errors). A recovery watchdog existed ONLY for
  the AEC path (`bufferCountValue < 3`); with AEC off (the default) nothing recovered a dead input.
- Fix (`Realtime/RealtimeAudio.swift`): added `scheduleNoAECInputWatchdog()` — 1.5s after a no-AEC
  start, if zero mic buffers have arrived, rebuild the input node and retry (up to 4 times). Wired
  into all no-AEC start sites (initial no-AEC start, AEC-failed fallback, AEC-starvation fallback).
- Build: `xcodebuild … -destination 'platform=macOS' build` → BUILD SUCCEEDED; relaunched the app.
- Note: this addresses the genuine no-buffer case seen in the log. If buffers arrive but are silent
  (granted=true), that's the separate TCC silence gotcha → `tccutil reset Microphone com.iris.app`
  + relaunch. A future hardening step: observe `AVAudioEngineConfigurationChange` to restart when
  another app grabs the audio device.

## 13:05 — [Realtime] Fix: sleeps mid-utterance + add create_folder tool
- Bug 1 (log: "idle 15s with no speech — pausing" while the user was still talking): the realtime
  idle timer was only reset on `speech_started` (once), so an utterance longer than the idle window
  tripped the pause mid-sentence. Fix (`RealtimeSession.swift`): track `userSpeaking` between
  server-VAD `speech_started`/`speech_stopped`; `resetIdleTimer` is a no-op while speaking, and the
  silence countdown (re)starts only on `speech_stopped`. Reset `userSpeaking` on connect/stop.
- Bug 2 ("create a folder called job hunter on my desktop" just opened Finder): there was no
  create-folder capability, so the model fell back to `open_folder`. Added `FolderOpener.create(
  name:in:)` (FileManager.createDirectory + reveal; reveals existing if present) and a
  `create_folder` realtime tool (name + optional location, defaults to Desktop) with a description
  that steers the model away from `open_folder` for "make/create". Location resolved via the
  existing `IntentRouter.resolveFolderArgument`.
- Build: `xcodebuild … -destination 'platform=macOS' build` → BUILD SUCCEEDED; relaunched.
